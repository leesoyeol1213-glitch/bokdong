# 임복동 세계일주 — Claude Code 이식 가이드

이 문서는 채팅 기반으로 개발해온 HTML 게임 "임복동 세계일주"를 Claude Code(VS Code) 환경으로 옮겨 이어서 개발하기 위한 인수인계 자료입니다.

---

## 1. 프로젝트 개요

- **게임명**: 임복동 세계일주 (한국어 방치형 모바일 게임)
- **컨셉**: 자전거로 세계일주하는 픽셀아트 아이들(idle) 게임. 한국 19개 도시 + 일본 5개 도시를 자전거로 여행
- **현재 버전**: v9.18
- **기술**: 순수 HTML/CSS/JavaScript 단일 파일 (프레임워크 없음, Canvas 2D 렌더링)
- **저장**: localStorage (키: `bkdng_v45` 게임데이터, `bkdng_sound` 사운드설정)
- **의미**: 사용자(비개발자 창업자)의 개발 여정 시작점이 된 프로젝트. 애착이 큼

---

## 2. 왜 이식하는가 (중요 맥락)

채팅으로 1.8MB 단일 HTML 파일을 통째로 주고받으며 개발해왔으나, 이 방식의 한계가 명확해졌습니다:

- 파일이 커서(1.8MB, 이미지 base64 포함) 전체를 한 번에 보기 어려움
- 한 곳을 수정하면 다른 곳이 깨지는 문제가 반복됨 (예: 목적지 변경 버그, 부스트 정렬 문제)
- 매 수정마다 전체 파일을 다시 받아 테스트해야 하는 비효율

**Claude Code로 옮기면**: 전체 코드베이스 직접 열람/검색, 파일 분리, git 버전관리, 실제 실행 테스트가 가능해집니다. 이게 이식의 목적입니다.

## 3. 파일 구조 (이 export 폴더)

```
bokdong_export/
├── index.html          # HTML 뼈대 (8KB) — 원래 단일파일에서 분리
├── src/
│   ├── game.js         # 게임 전체 로직 (255KB, 함수 156개)
│   └── style.css       # 스타일 (5KB)
├── assets/             # 이미지 49개 (1.2MB) — 원래 base64로 코드에 박혀있던 것
│   ├── cyc_1~5.png     # 복동이 일반 주행 스프라이트 (5프레임)
│   ├── b1_1~b5_5.png   # 부스트 스프라이트 (5단계 × 5프레임 = 25장)
│   ├── ty_1~10.png     # TY 수호천사 (10프레임)
│   └── bg_*.png        # 배경 이미지 9종 (서울/충주/제주/자연/해안/도시/달/신한/청학동)
└── CLAUDE_CODE_이식가이드.md  # 이 문서
```

### 원본과의 차이
- **원본**: 모든 CSS/JS/이미지가 `index.html` 하나에 base64로 인라인
- **분리본**: 이미지를 파일로 빼고 `img.src="./assets/xxx.png"`로 참조. 코드는 `img.src` 방식이라 경로만 바뀌면 그대로 동작함 (로직 수정 불필요)

### 실행 방법
로컬 서버 필요 (file:// 로는 이미지 CORS 문제 가능):
```bash
cd bokdong_export
python3 -m http.server 8000
# 브라우저에서 http://localhost:8000
```

## 4. 핵심 시스템 지도 (game.js)

### 데이터 구조 (파일 상단)
- `CITIES`: 도시 24개 (한국 19 + 일본 5). 각 도시는 `{n:이름, region:지역, bg:배경키, special:특수속성}`. 일본 도시는 `region:'일본'`
- `VEHS`: 자전거 30단계 + 로켓 1개. `{id:'v1'~'v30', n:이름, cat:'bike', sp:속도, cost:가격, km:요구주행거리, owned}`
- `ACHIEVEMENTS`: 업적 목록
- `OX_BY_CITY`, `CITY_DIST`, `TY_TALKS`: OX퀴즈, 도시간거리, 천사 대사

### 렌더링 (Canvas)
- `drawScene()`: 매 프레임 배경+캐릭터+UI 그리기. `frame++` 여기서 증가
- `drawBokdongSprite(ctx, cx, cy, scale, isRiding)`: 복동이 스프라이트 그리기. 부스트면 `b{tier}_{frame}`, 일반이면 `cyc_{frame}`
- `getBoostTier()`: 현재 자전거(v1~v30)가 부스트 5단계 중 몇 단계인지 (6개씩 묶음)
- `drawTY()`: TY 수호천사 (캐릭터 뒤 왼쪽 위를 따라다님, 10프레임 애니)
- `animLoop()`: 애니메이션 루프. 캐릭터 가로 이동(bikeX) 여기서 처리

### 게임 로직
- `tick()`: 1초마다 실행. 주행거리 누적, 도착 판정, 랜덤이벤트
- `enforceJapanRule()`: 일본은 부산 페리로만 진입 가능하도록 강제 (한국에서 일본 목적지면 취소)
- `autoPickNextDestination()`: 도착 후 다음 목적지 자동 선택
- `fireRandEvent()`: 라이딩 중 랜덤 이벤트 (킥라니 등)
- `load()` / 저장: localStorage 마이그레이션 포함

## 5. 현재 미해결 과제 (이식 후 최우선 작업)

### ★ 부스트 스프라이트 크기 변동 문제 (진행중)
**증상**: 부스트 켜면 복동이가 프레임마다 커졌다 줄었다 하고, 목이 잘림

**원인** (분석 완료): 원본 부스트 시트(`부스트_1~5단계.png`)가 **프레임마다 캐릭터 크기·위치·자세가 제각각**임. 일반 주행 시트(cyc)는 프레임 간 일관돼서 정상 작동하지만, 부스트 원본은 그렇지 않음. 자동 정렬(색 구분/바퀴/머리-발 검출)을 여러 방법으로 시도했으나 원본 자체의 불일치 때문에 모두 실패.

**권장 해결책**: 부스트 원본 시트를 **재생성**. 이미지 AI에 요청 시 핵심 프롬프트:
> "모든 프레임에서 캐릭터와 자전거의 위치·크기·카메라 거리를 100% 동일하게 고정. 다리(페달링)와 오라 이펙트만 변화. 캐릭터 머리 꼭대기와 자전거 바퀴 바닥이 모든 프레임에서 동일한 픽셀 위치에 오도록."

일반 cyc 시트가 잘 되는 이유가 이 일관성 때문임. 새 시트를 받으면 cyc와 동일한 방식(발끝 126px, 가로중심 64px, 128px 캔버스)으로 처리하면 됨.

### 기타 이식 후 검토사항
- 파일 분리했으니 모듈화 더 진행 가능 (data.js / render.js / logic.js 등으로 game.js 쪼개기)
- 이미지 스프라이트 아틀라스(sprite sheet) 방식으로 최적화 여지
- 사운드 시스템 (Web Audio API) 별도 파일 분리 가능

## 6. 개발 원칙 (사용자 요구 — 반드시 준수)

이전 개발에서 사용자가 강하게 요구한 원칙입니다. Claude Code에서도 지켜야 합니다:

1. **추측 금지**: 코드를 실제로 보고 작업. "아마 이럴 것"으로 수정하지 말 것
2. **한 곳만 고치고 "됐다" 금지**: 증상과 관련된 모든 코드 경로를 전수조사한 뒤 수정
3. **검증 후 보고**: 수정 후 실제로 동작하는지 확인(실행/테스트)하고 나서 완료 보고. 못 찾으면 솔직히 "못 찾았다"고 할 것
4. **정식 출시 기준**: 실제 출시하는 제품이라는 마음으로 품질 관리
5. **단계별 진행**: 큰 작업은 쪼개서, 확인받으며 진행

Claude Code 환경에서는 실제 실행/디버깅이 가능하므로 이 원칙들을 지키기 훨씬 쉬워집니다.

## 7. Claude Code 첫 시작 가이드

새 환경에서 이렇게 시작하면 됩니다:

1. 이 `bokdong_export/` 폴더 전체를 프로젝트 디렉토리로 사용
2. git 초기화: `git init && git add -A && git commit -m "이식: v9.18 from HTML"`
3. 로컬 서버 띄우고 브라우저에서 정상 작동 확인 (`python3 -m http.server 8000`)
4. 작동 확인되면 → 부스트 스프라이트 문제부터 (새 원본 시트 준비되면)
5. 이후 game.js 모듈 분리 등 리팩토링

Claude Code에게 첫 지시 예시:
> "bokdong_export 폴더가 HTML 게임을 이식한 거야. CLAUDE_CODE_이식가이드.md를 읽고, 먼저 로컬 서버로 게임이 정상 작동하는지 확인해줘. 그다음 부스트 스프라이트 문제를 보자."
